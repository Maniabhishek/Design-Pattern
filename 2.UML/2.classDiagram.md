  - then 2nd one is for Attributes
  - final one is for operations

- visibility markers
  - Public -> +
  - Private -> -
  - Protected -> #

### Perspective of class diagrams
- to product owner or any non tech guy
  - -> Customer -> places -> Order -> contains -> product (something like this)
  - -> Customer -> makes -> payment -> contains -> payment method
- to architects or Designers
  - specification perspective (focus is on the interfaces of the ADTs of software)
- Developers
  - implementation perspective (describes how classes will implement their interfaces)

### Relations between classes
- Association
- Aggregation
- Composition
- Inheritance
- Realization (Implementation)
- Dependency

#### Association
- this represents a "uses-a" relationship between two classes where one class uses or interacts with the other.
- user uses a order class, user uses a cart class, account uses wallet class

#### Aggregation
- this represents "has-a" relationship between two classes where one class uses or interacts with the other
- now why is this has a why can uses a well in the previous example if you see , users uses a cart , account uses a wallet wallet can different for different account ,if you destroy user cart will also go away , delete account wallet will go away but here in this case MenuItem class can be same for many restaurant , if you destroy or delete one restaurant still same Menuitem can be part of another restaurant

#### composite
- strong has-a relationship
- part cannot exist without the whole
- if the whole is destroyed then parts are also destroyed

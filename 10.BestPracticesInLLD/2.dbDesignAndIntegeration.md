### ER Diagrams
- An Entity relationship model represents entities, their attributes, and the relationships among them. Its the foundation for designing your db schema why it matters in LLD ?
- helps convert product requirements into data models
- bridges real world entities with database tables and object models

> Problem Statement: A user initiates a payment to a merchant, the payment might succeed or fail, success that means transaction, transaction may have refunds. Payments can be made via different methods

- how to design table from requirements
  - nouns = entity (user , merchant, paymentMethod, etc.,)
  - properties = columns (business_name, upi_id, etc)
  - verbs/actions = relationships

     - A transaction belongs to a payment request

### Mapping ER -> Class Model
- every table becomes class , every column becomes field and foreign keys becomes object references
```ts
class User {
    id: string;
    email: string;
    name: string;

    constructor(id: string, email: string, name: string) {
        this.id = id;
        this.email = email;
        this.name = name;
    }   
}

class Merchant {
    id: string;
    business_name: string;
    payment_requests: PaymentRequest[];
    constructor(id: string, business_name: string) {
        this.id = id;
        this.business_name = business_name;
        this.payment_requests = [];
    }
}

class PaymentRequest {
    id: string;
    amount: number;
    status: string;
    user: User;
    merchant: Merchant;

    constructor(id: string, amount: number, status: string, user: User, merchant: Merchant) {
        this.id = id;
        this.amount = amount;
        this.status = status;
        this.user = user;
        this.merchant = merchant;
    }
}

class Transaction {
    id: string;
    amount: number;
    status: string;
    payment_request: PaymentRequest;
    payment_method: PaymentMethod;

    constructor(id: string, amount: number, status: string, payment_request: PaymentRequest, payment_method: PaymentMethod) {
        this.id = id;
        this.amount = amount;
        this.status = status;
        this.payment_request = payment_request;
        this.payment_method = payment_method;
    }
}

class PaymentMethod {
    id: string;
    type: string;
    details: any;

    constructor(id: string, type: string, details: any) {
        this.id = id;
        this.type = type;
        this.details = details;
    }
}

class Refund {
    id: string;
    amount: number;
    status: string;
    transaction: Transaction;

    constructor(id: string, amount: number, status: string, transaction: Transaction) {
        this.id = id;
        this.amount = amount;
        this.status = status;
        this.transaction = transaction;
    }   
}
```

### What is DAO (Data Access Object)
- A DAO is a class that provides basic CRUD operations for a single entity table
- Purpose:
  - encapsulate new database access logic
  - keeps services clean sql/orm clutter
  - encourages seperation of concerns

```
class PaymentRequestDAO {
  save(paymentRequest: PaymentRequest): void
  findById(id: string): PaymentRequest
}
```
- does not support fetching related transaction or refund
- too low level for business flows
- any complex logic is being done by repository class

### Repository
- a repository is a domain centric that manages aggregates - not just on one table but related data grouped logically
- encapsulate complex joins and business rule
- provides high level, intent based APIs

```ts
class PaymentRepository{
  createPayment(userId, amount, status, merchantId, currency): void
  getPaymentDetails(id): Payment
  getPaymentHistoryForAUser(userid): Payment
}
```

### Real World enhancements and properties
- duplicate payment: use idempotency in payment request
- Refund must be atomic: wrap refund and status in a transaction block
- slow dashboards: Use pagination
- query explosions: fetch relation eagerly or use joins in repo
- scaling to millions: Indexing(Individual and Composite)
